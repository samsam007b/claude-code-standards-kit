# Contract — Transactional Email

> SQWR Project Kit contract module.
> Sources: RFC 5321 (SMTP), RFC 7489 (DMARC), RFC 8058 (List-Unsubscribe), Google Email Sender Guidelines 2024 (support.google.com/mail/answer/81126), react.email, Resend docs.

## Foundations

**Transactional email is a critical product feature.** A confirmation email landing in spam destroys the activation rate. A bounce rate >2% permanently degrades sending reputation.

Since **February 2024**, Google and Yahoo have made the following mandatory for senders of >5,000 emails/day:
1. SPF authentication
2. DKIM authentication
3. DMARC policy
4. One-click unsubscribe (RFC 8058)

**These requirements are recommended even below 5,000 emails/day** to protect deliverability from launch.

## 1. DNS Authentication — SPF, DKIM, DMARC

> Source: Google Email Sender Guidelines (support.google.com/mail/answer/81126)
> Source: RFC 7489 — DMARC (ietf.org/rfc/rfc7489)

### SPF — Sender Policy Framework

Declares which servers are authorized to send on behalf of your domain.

```dns
# TXT record on your domain (example with Resend)
example.com  TXT  "v=spf1 include:amazonses.com include:resend.com ~all"

# ~all = softfail (recommended — less aggressive than -all at launch)
# -all = hardfail (final target once domain is established)
```

### DKIM — DomainKeys Identified Mail

Cryptographic signature on each email. Verify that the provider (Resend, SendGrid, etc.) generates the keys and that you add the provided DNS records.

```dns
# Example generated by Resend (replace with actual values)
resend._domainkey.example.com  CNAME  resend.domainkey.net
```

### DMARC — Domain-based Message Authentication

Policy that tells servers what to do if SPF/DKIM fail.

```dns
# TXT record — progressive configuration
_dmarc.example.com  TXT  "v=DMARC1; p=none; rua=mailto:dmarc@example.com"

# DMARC deployment steps:
# 1. p=none  → monitoring only (0 impact on deliverability)
# 2. p=quarantine → suspicious emails to spam (after reviewing reports)
# 3. p=reject → non-compliant emails rejected (final goal)
# rua = address to receive aggregate reports
```

**Verification:**
```bash
# Verify SPF
nslookup -type=TXT example.com

# Verify DMARC
nslookup -type=TXT _dmarc.example.com

# Online tool: MXToolbox (mxtoolbox.com/SuperTool.aspx)
```

## 2. Provider — Resend (recommended for Next.js)

> Source: Resend docs (resend.com/docs), React Email (react.email/docs)

```bash
npm install resend @react-email/components
```

```typescript
// lib/email.ts
import { Resend } from 'resend'

const resend = new Resend(process.env.RESEND_API_KEY)

export async function sendTransactionalEmail({
  to,
  subject,
  react,
}: {
  to: string
  subject: string
  react: React.ReactElement
}) {
  const { data, error } = await resend.emails.send({
    from: 'App Name <noreply@example.com>',  // domain verified in Resend
    to,
    subject,
    react,
  })

  if (error) {
    // Log the error but do not expose it to the client
    console.error('[email] Send failed:', error)
    throw new Error('Email send failed')
  }

  return data
}
```

**Recommended sending domains:**
```
Transactional  → noreply@example.com     (confirmations, MFA, invoices)
Marketing      → hello@mail.example.com  (campaigns, newsletters)
Support        → support@example.com     (tickets)

Rule: marketing sending domain separate from transactional domain.
A bounce on the newsletter must not penalize confirmation emails.
```

## 4. Next.js API Route — Sending

```typescript
// app/api/auth/confirm/route.ts
import { NextRequest, NextResponse } from 'next/server'
import { sendTransactionalEmail } from '@/lib/email'
import WelcomeEmail from '@/emails/WelcomeEmail'

export async function POST(request: NextRequest) {
  const { email, userName, confirmUrl } = await request.json()

  // Mandatory validation before sending (input sanitization)
  if (!email || !email.includes('@')) {
    return NextResponse.json({ error: 'Invalid email' }, { status: 400 })
  }

  try {
    await sendTransactionalEmail({
      to: email,
      subject: `Confirmez votre compte`,
      react: WelcomeEmail({ userName, confirmUrl }),
    })
    return NextResponse.json({ success: true })
  } catch {
    return NextResponse.json({ error: 'Email failed' }, { status: 500 })
  }
}
```

## 5. Bounce and Unsubscribe Handling

> Source: RFC 8058 — List-Unsubscribe (ietf.org/rfc/rfc8058)
> Source: GDPR Article 6 — legal basis for processing

### Bounces

| Type | Meaning | Action |
|------|---------|--------|
| **Hard bounce** | Non-existent address | Remove immediately from the list |
| **Soft bounce** | Full mailbox, server temporarily unavailable | Retry 3× over 48h, then remove |

**Critical threshold: bounce rate <2%.** Beyond this, sending reputation degrades and Gmail begins rejecting emails.

```typescript
// Resend webhook to handle bounces automatically
// app/api/webhooks/resend/route.ts
export async function POST(request: NextRequest) {
  const event = await request.json()

  if (event.type === 'email.bounced') {
    // Mark email as invalid in the database
    await markEmailAsBounced(event.data.email_id)
  }

  return NextResponse.json({ received: true })
}
```

### Unsubscribe (mandatory — RFC 8058 + Google 2024)

Resend handles `List-Unsubscribe` automatically if configured. For marketing emails (not transactional):

```typescript
await resend.emails.send({
  from: 'hello@mail.example.com',
  to,
  subject,
  react: NewsletterEmail({ ... }),
  headers: {
    'List-Unsubscribe': '<https://example.com/unsubscribe?token=...>',
    'List-Unsubscribe-Post': 'List-Unsubscribe=One-Click',  // RFC 8058
  },
})
```

## 6. Deliverability Thresholds

> Source: Google Postmaster Tools guidelines, SendGrid Email Deliverability Guide (2024)

| Metric | Acceptable threshold | Action if exceeded |
|--------|---------------------|-------------------|
| **Bounce rate** | <2% | Clean the list immediately |
| **Spam rate** | <0.1% | Review content and opt-ins |
| **Unsubscribe rate** | <0.5% | Review frequency and targeting |
| **Open rate** (transactional) | >50% | Below: check deliverability |
| **Open rate** (marketing) | >25% | Below: rework subject/segment |

**Monitor with:** Google Postmaster Tools (postmaster.google.com) — free, data on domain reputation.

## Pre-launch Email Checklist

### Blockers

- [ ] SPF configured on the sending domain
- [ ] DKIM configured (DNS records provided by the provider)
- [ ] DMARC configured (`p=none` at minimum for monitoring)
- [ ] Domain verified in Resend/SendGrid
- [ ] Template tested on Gmail, Outlook, Apple Mail (Litmus or Resend preview)
- [ ] Unsubscribe link present in marketing emails
- [ ] `RESEND_API_KEY` in environment variables (never hardcoded)

### Important

- [ ] Transactional domain separate from marketing domain
- [ ] Bounce webhook configured → automatic removal of hard bounces
- [ ] Google Postmaster Tools configured for monitoring
- [ ] DMARC moved to `p=quarantine` after 2 weeks of monitoring
- [ ] Confirmation email tested end-to-end (click → account activation)

### Desirable

- [ ] DMARC `p=reject` (final goal)
- [ ] BIMI configured (logo in the inbox — Gmail/Apple Mail)
- [ ] Litmus test across 15+ email clients
